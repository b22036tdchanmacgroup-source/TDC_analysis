이슈 분석

이관 시뮬레이션 및 시나리오 테스트에서 발견된 문제

발견된 이슈 목록
1
문서 접근 비용 과다
긴급
시나리오 테스트 결과, 역할·목적에 따라 동일 문서 도달에 클릭 수가 과도. 특히 부서 경계를 넘는 접점 문서는 위치 예측 자체가 어려움
2
이관 시 빈 폴더 다수 발생
긴급
일률적 템플릿을 전체 S/W에 강제 적용 시 문서가 없는 카테고리에 빈 폴더 생성 → 구조 신뢰도 저하, 사용자 혼란 가중
3
문서명 불명확 → 귀속 판단 어려움
중요
기존 문서명에 내용 유추 정보 부족 + 동일 성격 문서의 다른 이름 병존 → 이관 시 분류 판단 불가 사례 다수
4
엔지니어↔개발자 접점 문서 귀속 기준 없음
중요
기능 명세서, 알고리즘 설계서 등 양쪽이 함께 참조하는 문서의 귀속 폴더가 불명확 → 이관 시 개인 판단에 의존
5
이관 후 구조 유지 보장 없음
중요
구조를 바꿔도 임직원이 새 기준으로 문서를 생성·업로드하지 않으면 시간이 지나며 재파편화 발생. 정착 메커니즘 부재

S/W 분류 체계

건설분야 S/W 전체를 4가지 유형으로 분류한 기준과 배정 현황

분류 원칙 및 기준
분류 판단 기준 3가지
①
주된 문서 발생 단계 — 엔지니어링 설계 중심인지, UX·화면 중심인지, 해석·엔진 중심인지
②
주요 사용자 — 엔지니어(도메인 로직)가 주도하는지, 개발자(기술구현)가 주도하는지
③
산출물 성격 — 설계 계산서·도면이 핵심인지, UI/UX 산출물이 핵심인지, 엔진·라이브러리가 핵심인지
경계 S/W 처리 원칙
복수 유형에 걸치는 S/W는 주된 사용 목적과 문서 발생 비중이 높은 단계를 기준으로 하나의 유형으로 확정합니다.

보조적 성격의 카테고리는 C-3 기능명세서 또는 해당 단계 서브카테고리로 흡수합니다. 유형을 복수로 허용하면 관리 복잡도가 급격히 올라가기 때문에 단일 배정이 원칙입니다.

※ 최종 유형은 각 S/W 담당팀 인터뷰 후 확정 예정

TYPE C

전체 S/W 유형 배정 현황
⚠ 사내 전용 S/W (SW 사용 범위가 대외 서비스가 아닌 내부 업무용인 경우) 및 R&D 과제·BEPs는 이번 분류 범위에서 제외합니다. 순수 건설분야 기술 S/W를 우선 대상으로 합니다.

이관 샘플 테스트 — 발견된 구조 이슈
①
빈 폴더 다수 발생
일률 템플릿 적용 시 S/W마다 실제 문서 없는 카테고리에 빈 폴더 생성. 예) WayDraw에 "B. 엔지니어링 설계" 폴더, BridgePlanner에 "B. UX·화면설계" 폴더
✓ 해결 — 유형별 최적화 카테고리 적용, 문서 존재 시에만 폴더 생성 원칙
②
엔지니어↔개발자 접점 문서 귀속 불명확
기능 명세서, 알고리즘 설계서 등 양쪽이 참조하는 문서의 폴더 귀속 판단 불가 → 이관 작업자마다 다른 폴더에 배치
✓ 해결 — C-3. 기능명세서·알고리즘 설계서 서브카테고리로 귀속 고정
③
현장 성과품 문서의 귀속 위치 모호
대산-당진 현장의 설계도·수량산출서 등이 S/W 탭 안에서 어느 단계(B.설계? E.검수?)에 귀속해야 하는지 불명확. 같은 현장이 6개 S/W에 분산
→ 별도 검토 — 성과품 분류 방식 2가지 비교 후 결정
④
Z. 작업장의 성격 재정의 필요
Z.작업장(팀별 임시 작업 공간)으로 확정. 기능명세서·알고리즘 설계서 등 접점 문서는 C-3으로 귀속. 카테고리별 문서 축적 지침을 사전에 팀간 공유하는 것이 운영의 전제 조건.
✓ 해결 — Z-1.엔지니어링팀 / Z-2.개발팀 / Z-3.디자인팀 작업장으로 확정

폴더 체계 설계

4 Depth 구조 내 S/W 중심 분류 원칙 및 작업 순서

Depth 구조 정의
Depth 0 최상단 (회사 고정) TDC / GPD 구분
↓
Depth 1 탭 (S/W명 또는 도메인) BridgePlanner, 천지인…
↓
Depth 2 카테고리 (업무 흐름 단계) A.기획·조사 / B.설계 …
↓
Depth 3 서브카테고리 (세부 분류) A-1. 요건정의 / A-2. 시장조사…
3대 설계 원칙
원칙 1 — 문서 있을 때만 폴더 생성
공통 뼈대는 가이드, 강제 아님. 해당 S/W에 실제 문서가 없는 카테고리는 생성하지 않음 → 빈 폴더 방지
원칙 2 — 카테고리는 업무 흐름 기준
조직·부서가 아닌 기획→설계→개발→검수→운영의 흐름으로 구성. 엔지니어·개발자가 동일 폴더 내 각 단계에 업로드
원칙 3 — Z. 작업장 카테고리 고정
모든 S/W 폴더에 Z. 작업장 고정. 팀별 임시 작업 공간으로 작업 완료 문서는 A~E 해당 카테고리로 이동. 엔지니어·개발자·디자인팀이 카테고리별 축적해야 할 문서 유형을 사전에 지침으로 정리해 공유함으로써 협업 기반을 만드는 것이 전제가 되어야 함의
Z. 작업장 — 운영 원칙 및 카테고리별 문서 축적 지침
전제 조건: 엔지니어·개발자·디자인팀이 각 카테고리(A~E)에 어떤 문서를 축적해야 하는지 사전에 지침으로 정리해 공유해야 합니다. 지침 없이 이관하면 작업장이 영구 보관소로 전락해 기존 문제가 반복됩니다.
Z. 작업장 구성 (전 유형 공통)
Z-1
엔지니어링팀 작업장 — 설계 진행 중 초안, 검토 중 도면, 미확정 계산서
Z-2
개발팀 작업장 — 개발 진행 중 코드 스펙, 미완성 기능명세서, 이슈 임시 기록
Z-3
디자인팀 작업장 — UI 시안 초안, 검토 중 와이어프레임, 미확정 스타일가이드
금지: 작업 완료 문서를 Z.작업장에 방치 금지. 완료 즉시 A~E 해당 카테고리로 이동.
카테고리별 문서 축적 지침 (예시)
카테고리 엔지니어링팀 개발팀
A. 기획·조사 시장조사, 기술동향, 기획서 기술 검토서, 스택 분석
B. 설계 (유형별 상이) 알고리즘, 계산근거, 도메인로직 시스템 아키텍처, DB 설계
C. 기능 요건·명세 UI 요건, 기능명세서(작성) 기능명세서(검토), API 정의서
D. 개발·구현 테스트 케이스, 검수 기준 개발계획, 매뉴얼, 릴리즈노트
E. 검수·품질 현장 검증, 성과품, 홍보자료 배포이력, 테스트 결과
※ 위 예시는 초안. 담당팀 인터뷰를 통해 S/W별로 확정 필요.
접점 문서 처리 원칙: 기능명세서·알고리즘 설계서처럼 엔지니어↔개발자 양측이 참조하는 문서는 Z.작업장이 아닌 C. 기능 요건·명세 > C-3. 기능명세서·알고리즘 설계서로 귀속. Z.작업장은 작업 완료 전 임시 보관 공간으로만 사용.

1

설계도·계산서·수량산출서만 성과품인지, 회의록·공문·행정서류도 포함인지 기준이 없으면 작업자마다 판단이 달라짐
2
bCMf Raw data 별도 정의
bCMf의 현장 문서는 설계 성과품이 아닌 공사 중 생산 데이터(일일보고·품질·안전 기록). 다른 S/W 성과품과 성격이 달라 명칭과 관리 기준을 별도로 정의해야 함
3
현장 추가 기준 필요
